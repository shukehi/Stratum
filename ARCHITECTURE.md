## 一、第一性原理：市场的底层真相

### 市场本质

```
市场 = 一台持续运行的拍卖机器

唯一工作：
  1. 价格发现 — 寻找买卖双方都愿意成交的价格
  2. 流动性分配 — 为大型机构订单寻找足够的对手方

推论：
  价格的每一次移动，要么是在"寻找流动性"，
  要么是在"远离不公平价格回归公允价值"。
  没有第三种原因。
```

### 信号的定义

一个有效的交易信号必须能回答三个问题：

| 问题 | 对应层 | 工具 |
|------|-------|------|
| 价格处于什么状态？（公允/溢价/折价） | 市场机制层 | OI、资金费率、Volume Profile |
| 价格会向哪里移动？（去哪里取流动性） | 结构层 | 流动性地图、平仓热力图 |
| 现在是否真的在移动？（有没有真实的力） | 订单流层 | CVD、Delta吸收 |

## 二、分析框架：四层架构

```
┌─────────────────────────────────────────────────────┐
│  Layer 4: 入场触发（When — 现在进？）                 │
│  订单流确认: CVD方向 / Delta吸收 / 成交量异常          │
├─────────────────────────────────────────────────────┤
│  Layer 3: 结构设置（Where — 在哪里进？）              │
│  流动性地图: 止损聚集区 / 等高等低 / 平仓热力图        │
├─────────────────────────────────────────────────────┤
│  Layer 2: 价格公允区（Context — 现在贵还是便宜？）     │
│  成交量分布: VPOC / 价值区间(VAH/VAL) / HVN / LVN    │
├─────────────────────────────────────────────────────┤
│  Layer 1: 市场机制（Regime — 谁在驱动市场？）         │
│  OI + 资金费率 + 现货溢价 + 去杠杆检测                │
└─────────────────────────────────────────────────────┘
```

### Layer 1 — 市场机制（✅ 已实现）

| 指标 | 含义 | 当前实现 |
|------|------|---------|
| OI上升 + 资金费率偏高 | 杠杆多头主导，扫多头止损风险高 | ✅ `detect-market-regime.ts` |
| OI下降 + 资金费率回归 | 去杠杆，避免追势 | ✅ DELEVERAGING_VACUUM |
| 现货溢价 | 真实买压 vs 纯期货投机 | ✅ `analyze-participants.ts` |

### Layer 2 — 价格公允区（❌ 未实现 → PHASE_17目标）

**Volume Profile**（成交量分布）是这一层的核心工具：

```
VPOC（成交量中心价）
  = 某时段内成交量最大的价格点
  = 机构认为的"公允价值"
  = 价格倾向于回归的磁力中心

价值区间（Value Area）= 容纳70%成交量的价格区间
  VAH（价值区间上限） = 上方自然阻力
  VAL（价值区间下限） = 下方自然支撑

HVN（高成交量节点）= 价格在此被大量吸收 → 强支撑/阻力
LVN（低成交量节点）= 价格在此快速穿越 → 价格真空带，预期快速移动
```

**日线偏向判断（当前实现 vs 目标）**：

```
当前（PHASE_16 市场结构法）：
  摆高摆低序列 → HH/HL = bullish, LH/LL = bearish
  优点：比EMA更接近结构  缺点：仍依赖时间框架，缺少成交量支撑

目标（PHASE_17 Volume Profile法）：
  日线VPOC + 价值区间 → 价格在VAH上方=溢价=偏空, VAL下方=折价=偏多
  优点：基于真实成交量，公允价值有机构行为支撑
```

### Layer 3 — 结构设置（⚠️ 部分实现）

| 工具 | 含义 | 当前状态 |
|------|------|---------|
| FVG（公允价值缺口） | 机构急速建仓留下的价格效率不足区域 | ✅ 已实现 |
| 流动性扫描 | 价格突破止损聚集区后快速反转 | ✅ 已实现 |
| 摆高摆低 | 机构订单位置的近似标记 | ✅ 已实现 |
| 等高/等低（Equal H/L） | 两个以上相同价位的高/低点，止损极度聚集 | ❌ 未实现 |
| 平仓热力图 | 哪些价位的杠杆仓位将被强制平仓 | ❌ 未实现 |

### Layer 4 — 入场触发（❌ 未实现 → PHASE_18目标）

**CVD（累计成交量差）**：

```
CVD = Σ(主动买入量) - Σ(主动卖出量)

判断逻辑：
  价格上涨 + CVD上涨 → 真实买压驱动 ✅ 信号可信
  价格上涨 + CVD下降 → 被动做市商推价，无真实买压 ❌ 假突破，过滤

Delta吸收（Absorption）：
  价格不动 + CVD持续下降 → 有人在静默吸收卖单 → 机构建仓 → 价格即将上涨

近似计算（无Tick数据时的Kaufman近似法）：
  delta_per_candle = (close - open) / (high - low) × volume
  accuracy ≈ 70%，适用于中期（4h）信号过滤
```

## 四、开发路线图

### PHASE_17 — Volume Profile（成交量分布分析）

**目标**: 用真实成交量分布替代时间框架相关的趋势判断

**核心工具**:
- `computeVolumeProfile(candles, buckets)` → VPOC + 价值区间 + HVN/LVN
- `detectDailyBiasVP(candles1d)` → 基于VPOC/VAH/VAL的日线偏向

**判断逻辑**:
```
价格 > VAH（价值区间上限）→ 溢价区 → bearish（偏空，价格倾向回归）
价格 < VAL（价值区间下限）→ 折价区 → bullish（偏多，价格倾向回归）
价格在 VAL ~ VAH 内       → 均衡区 → neutral（双向机会）
价格在 HVN 附近           → 强磁吸区，预期震荡吸收
价格在 LVN 内             → 真空带，预期快速穿越
```

**所需数据**: 已有OHLCV → 无需新数据源

**文件变更**:
```
新增:
  src/domain/market/volume-profile.ts        — VolumeProfile类型定义
  src/services/analysis/compute-vp.ts        — Volume Profile计算
  src/services/regime/detect-bias-vp.ts      — 基于VP的日线偏向检测

修改:
  src/domain/market/daily-bias.ts            — 替换EMA字段为VP字段
  src/services/regime/detect-daily-bias.ts   — 调用VP方法
  src/app/config.ts                          — 添加VP相关参数
```

**测试要求**:
- `computeVolumeProfile` — VPOC精度验证、价值区间70%验证
- `detectDailyBiasVP` — 溢价/折价/均衡三种场景
- 与现有`evaluate-consensus.ts`集成不破坏现有测试

**估时**: 人工 3天 / CC 30分钟
**优先级**: ⭐⭐⭐⭐⭐（最高）

### PHASE_18 — CVD订单流确认层

**目标**: 在FVG和流动性扫描信号上叠加订单流确认，过滤假信号

**核心工具**:
- `computeApproxDelta(candle)` → 单根K线的近似Delta（Kaufman法）
- `computeCVD(candles)` → 累计成交量差序列
- `detectOrderFlowBias(candles, window)` → 近N根K线的CVD趋势

**过滤规则**:
```
FVG做多信号 + CVD近期上升 → 信号保留
FVG做多信号 + CVD近期下降 → 降级（COUNTER_FLOW标记）

流动性扫描多头 + 扫描后CVD快速上升 → 高置信度
流动性扫描多头 + 扫描后CVD继续下降 → 假扫描，过滤
```

**新增ReasonCode**:
```typescript
"ORDER_FLOW_ALIGNED"   // CVD与信号方向一致，置信度提升
"ORDER_FLOW_COUNTER"   // CVD与信号方向相反，信号降级
```

**文件变更**:
```
新增:
  src/domain/market/order-flow.ts            — OrderFlow类型定义
  src/services/analysis/compute-cvd.ts       — CVD/Delta计算
  src/services/analysis/detect-order-flow.ts — 订单流偏向检测

修改:
  src/services/consensus/evaluate-consensus.ts — 新增订单流门槛
  src/domain/common/reason-code.ts             — 新增ORDER_FLOW_*
```

**估时**: 人工 3天 / CC 30分钟
**优先级**: ⭐⭐⭐⭐

### PHASE_19 — 等高等低（Equal Highs/Lows）检测

**目标**: 识别止损高度聚集的"陷阱"价位，作为更精准的流动性目标

**核心逻辑**:
```
等高（Equal Highs）：
  最近N根K线中，有2根以上的最高价在容差范围内（如0.1%）
  → 做空止损聚集 = 买方流动性池
  → 价格极可能上扫后快速反转

等低（Equal Lows）：
  最近N根K线中，有2根以上的最低价在容差范围内
  → 做多止损聚集 = 卖方流动性池
  → 价格极可能下扫后快速反转

与普通摆高摆低的区别：
  普通摆高 = 单个局部最高点（1个止损聚集）
  等高     = 多个相同价位（止损极度密集，机构更感兴趣）
```

**估时**: 人工 2天 / CC 20分钟
**优先级**: ⭐⭐⭐

---

### PHASE_20 — 平仓热力图集成

**目标**: 接入真实的杠杆仓位分布数据，识别价格的真实"磁力区"

**数据来源**:
- Coinglass API（`https://open-api.coinglass.com`）
- 提供BTC/ETH的多空仓位分布和预估平仓价位

**核心价值**:
```
平仓热力图 = 哪些价位有大量杠杆仓位将被强制平仓

应用场景：
  信号止盈位 = 最近的高密度平仓区（价格被吸引到这里）
  避开低密度区域发出信号（没有足够流动性支撑移动）
  大型平仓区被扫 = 极高置信度反转信号
```

**估时**: 人工 3天 / CC 45分钟
**优先级**: ⭐⭐⭐

---

### PHASE_21 — 事件驱动架构（长期目标）

**目标**: 从"定时扫描"升级为"价格触达关键位即触发扫描"

**当前架构（时间驱动）**:
```
每4小时 → 扫描所有symbol → 评分 → 发出信号
问题：信号可能延迟最长4小时；大量无效扫描
```

**目标架构（事件驱动）**:
```
持续监听价格 → 价格触达关键位（VPOC/流动性区/等高等低）→ 立即触发深度分析
优点：零延迟；只在有意义的时候分析；更接近机构行为
```

**估时**: 人工 2周 / CC 3小时
**优先级**: ⭐⭐（长期，当前定时扫描够用）

## 五、设计约束

### 不应该做的事

```
❌ 增加更多K线时间框架（15m、周线）
   → 更多时间框架 = 更多冲突信号，不是更多信息
   → 正确做法是用Volume Profile替代多时间框架

❌ 增加传统技术指标（RSI、MACD、布林带）
   → 都是价格的数学变换，进一步远离第一性原理
   → 有意义的指标必须能解释"为什么"，不只是"是什么"

❌ 用机器学习预测价格方向
   → 市场机制会变化，历史数据会过拟合
   → ML适合用于执行优化（滑点预测、最优入场时机），不适合方向预测

❌ 将信号质量与K线形态绑定
   → 锤子线、十字星等形态是统计相关，无因果机理
   → 一切信号质量判断必须基于流动性、成交量或订单流
```

### 数据分层原则

```
第1层（最高价值）: 订单簿实时数据、Tick成交数据
  → 对4h信号意义有限，实现成本极高

第2层（高价值）: 成交量分布（可从OHLCV计算）、CVD（Kaufman近似）
  → PHASE_17/18 目标，已有数据可计算

第3层（中价值）: K线结构（FVG、流动性扫描）
  → 当前系统核心，继续保留

第4层（低价值）: 时间框架相关技术指标（EMA、RSI等）
  → 已移除EMA，继续清理此类依赖
```
