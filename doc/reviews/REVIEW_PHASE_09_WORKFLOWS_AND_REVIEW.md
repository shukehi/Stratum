# Review Phase 09: Workflows And Review

## 1. 复审目标

确认端到端工作流按第一性原理顺序执行，且失败恢复与日志可支撑真实研究闭环。

## 2. 复审范围

- `src/workflows/*`
- `src/app/scheduler.ts`
- `src/index.ts`
- 对应集成测试

## 3. 工程复审问题

1. `runAnalysisCycle()` 顺序是否正确（含现货价格拉取、时段识别、基差计算、confluence 检测、入场确认）。
2. `runBackfill()` 是否只做历史回填而不混入策略判断。
3. `runPaperTradeReview()` 是否具备最小复盘能力。
4. `runCalibrationReview()` 是否在样本量达标后可运行，且不自动修改配置。
5. 日志是否覆盖关键节点。
6. 单模块失败是否可恢复。

## 4. 第一性原理复审问题

1. 主流程是否保持 `state -> participants -> structure -> confirmation -> risk -> macro -> persist -> alert`。
2. 是否存在工作流里重写业务规则的情况。
3. 是否在工作流层绕过某个关键层。
4. 复盘流程是否能回溯”为什么当时发了这个信号”。
5. 校准工作流是否能验证 RR 阈值、时段修正和 confluence 加分的实际效果。
6. 校准报告是否只作为建议而非自动执行。

## 5. 常见失败模式

- 工作流跳过状态层或参与者层
- 工作流先发告警后写库
- 工作流中重新实现共识逻辑
- 复盘只看盈亏，不保留触发上下文
- 校准工作流自动修改配置，绕过人工审核
- 工作流中遗漏现货价格拉取或时段识别步骤

## 6. 通过标准

- 主流程顺序与主文档一致
- 工作流只负责编排，不重写核心业务规则
- 端到端日志和复盘足以支撑研究闭环
- 校准工作流可运行且只输出建议
