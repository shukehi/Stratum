# Stratum 第一性原理评判标准

## 1. 文档目标

本文档是 Stratum 项目的持续评判标准，用于在开发、测试和运行三个阶段回答同一个问题：

**系统是否真正从"谁被迫交易"出发做判断，还是退回到了"找图形 → 出信号"的模式？**

本文档不是一次性检查清单。它应该在以下时刻被使用：

- 每完成一个开发阶段后的自审
- 代码复审时的对照依据
- 系统上线运行后的定期回顾
- 任何阈值调整前的前提验证

## 2. 核心判定原则

> 如果你能对系统发出的每一个信号和每一个"不交易"决定，都沿着因果链追溯到"谁被迫交易"或"没有人被迫交易"，那它就是第一性原理系统。如果追溯到某一层就断了——"不知道为什么到这一步"——那它就不是。

因果链：`市场状态 → 参与者压力 → 结构位置 → 风险收益 → 语义校验 → 告警`

## 3. 第一层：因果链完整性（开发阶段）

这一层检查的是**系统有没有在做正确的事**。在每个开发阶段完成后执行。

### 3.1 去掉结构层后，系统还能输出什么？

**检验方法**：将 `detectStructuralSetups()` 替换为永远返回空数组的 stub，运行系统。

**预期结果**：系统仍然能输出有意义的 `MarketContext`——包含正确的 regime、参与者压力方向、基差背离状态、流动性时段。

**如果不符合**：说明因果链的前两层（状态 + 参与者）不是独立的，系统实际上是结构驱动的，前两层只是装饰。

**对应主文档**：§3A.1, §3A.2, §4.1, §4.2

### 3.2 去掉 LLM 后，系统还能正常工作吗？

**检验方法**：将 `applyMacroOverlay()` 替换为永远返回 `{ action: "pass" }` 的 stub，运行系统。

**预期结果**：信号质量基本不变。可能少了极端事件保护，但核心信号逻辑不受影响。

**如果不符合**：说明 LLM 实际上进入了核心决策路径，违反了 §3A.5 的认识论约束。

**对应主文档**：§3A.5, §4.5, §6.6

### 3.3 把判断链顺序反过来会怎样？

**检验方法**：修改 `runAnalysisCycle()`，让结构层先跑，然后用 regime 和参与者做过滤。对比正常顺序和反转顺序的输出。

**预期结果**：反转顺序应该导致完全不同的候选集合。正常顺序下被 regime 或参与者层前置过滤掉的场景，在反转顺序下会进入结构层并产生候选。

**如果不符合**：说明前两层没有真正约束结构层——它们只是在做事后审批，而不是事前框定。

**对应主文档**：§3A.3, §4.1, §10

### 3.4 系统能否解释"为什么不交易"？

**检验方法**：随机选取 10 个没有产生信号的分析周期，检查日志或数据库记录。

**预期结果**：每个"不交易"决定都有结构化的跳过原因：

- `regimeConfidence` 低于阈值
- 参与者真空（去杠杆刚完成）
- `structureScore` 不够
- `riskReward` 不达标
- `confirmationStatus` 仍为 `pending`
- 当前处于不可交易时段或事件窗口

**如果不符合**：如果跳过时只有笼统的"no signal"，说明系统没有真正走完因果链就放弃了。第一性原理系统的"不交易"和"交易"一样重要，都必须有完整的推理记录。

**对应主文档**：§15A.6, §20

## 4. 第二层：行为真实性（测试阶段）

这一层检查的是**系统有没有在正确地做事**。通过构造特定测试用例验证。

### 4.1 参与者层是否区分"新开仓"和"平仓驱动"？

**测试用例**：价格持续上涨，但 OI 持续下降。

**预期结果**：系统识别为"空头回补驱动"（deleveraging），而非"新多头趋势"。`pressureType` 不应输出 `squeeze-risk`，`bias` 不应简单输出 `long-crowded`。

**如果不符合**：说明参与者层把"价格涨 + OI 跌"错误地解释成看多信号，退回到了"价格往哪走"的思维。

**对应主文档**：§3A.2, §15A.3（去杠杆而非新开仓）

### 4.2 基差背离是否正确增强 squeeze/flush 判断？

**测试用例**：funding rate 持续为负，同时 `spotPrice > perpPrice`（现货溢价）。

**预期结果**：`basisDivergence = true`，`squeeze-risk` 置信度上调，附加 `PARTICIPANT_BASIS_DIVERGENCE` reason code。

**如果不符合**：说明系统只看合约端单边信息，没有交叉验证现货端和合约端的参与者行为不一致。

**对应主文档**：§15A.3（现货-永续基差背离）

### 4.3 同一结构在不同流动性时段的评分是否不同？

**测试用例**：构造两个结构完全相同的 FVG setup，一个时间戳在 UTC 03:00（亚盘 `asian_low`），一个在 UTC 07:00（伦敦 `london_ramp`）。

**预期结果**：亚盘的 `structureScore` 被 `sessionDiscountFactor`（默认 0.8）折扣，伦敦的被 `sessionPremiumFactor`（默认 1.1）加成。

**如果不符合**：说明系统没有考虑流动性时段对虚假突破概率的影响，所有时段等价处理。

**对应主文档**：§15A.1A

### 4.4 触碰入场区域但无拒绝信号时，是否仍为 pending？

**测试用例**：价格进入 `entryLow ~ entryHigh` 区域，但没有出现下影线 ≥ `confirmationShadowRatio` 或"不再创新低"的确认信号。

**预期结果**：`confirmationStatus = "pending"`，信号等级最高为 `watch`。不产生 `standard` 或 `high-conviction` 信号。

**如果不符合**：说明确认机制没有生效，系统退回到"触碰即触发"的逻辑，违反了 §3A.4 的因果约束。

**对应主文档**：§4.3B, §15A.5

### 4.5 复合结构是否比单一结构获得更高评分？

**测试用例**：构造三个 setup：
- Setup A：仅有 FVG，无其他结构重叠
- Setup B：FVG + swing-low 在同一价格区域
- Setup C：FVG + swing-low + liquidity-pool 在同一价格区域

**预期结果**：`structureScore` 排序为 C > B > A，且 B 比 A 高至少 `confluenceBonus`（默认 +10），C 比 A 高至少 +15。

**如果不符合**：说明系统忽略了 confluence，所有结构同权处理。

**对应主文档**：§4.3A, §15A.5

### 4.6 LLM Prompt 是否遵守信息边界？

**测试用例**：构造一个完整的候选信号，检查 `buildPrompt()` 的输出字符串。

**预期结果**：Prompt 中包含新闻、价格、方向、等级、contextReason，但不包含 `structureScore`、`stopLoss`/`takeProfit` 具体价格、`participantConfidence`、`confluenceFactors`、原始 K 线或 OI 数据。

**如果不符合**：说明 Prompt 暴露了因果链内部判断，LLM 有机会反过来"评判"因果链，违反了 §3A.5 的认识论边界。

**对应主文档**：§14.1

### 4.7 downgrade 和 block 的行为是否正确？

**测试用例**：

| 输入 | 预期输出 |
|------|---------|
| `high-conviction` + `downgrade` | `standard` |
| `standard` + `downgrade` | `watch` |
| `watch` + `downgrade` | `watch`（不丢弃） |
| 任何等级 + `block` | 写入数据库（`alert_status = "blocked_by_macro"`），不发告警 |
| 任何等级 + `pass` | 保持原等级，`catalystSummary` 附加到告警文案 |

**如果不符合**：说明降级幅度错误（直接降到最低而非降一级），或 block 信号被丢弃导致研究样本丢失。

**对应主文档**：§14.2, §14.3

## 5. 第三层：认知诚实性（运行阶段）

这一层检查的是**系统有没有对自己诚实**。这是最难也是最重要的一层。需要系统运行一段时间后才能评估。

### 5.1 "不交易"的比例是否合理？

**检验方法**：统计过去 N 个分析周期中，产生信号 vs 没有产生信号的比例。

**预期结果**：大多数分析周期应该输出"不交易"。一个第一性原理系统在大多数时间应该认为市场不具备交易条件——因为大多数时间市场状态不够清晰、或参与者压力不明确、或结构位置不在低风险区域。

**警告阈值**：如果超过 30% 的分析周期都产生信号，需要审查过滤层的阈值是否太松。如果低于 5%，需要审查是否太严导致有效机会被过滤。

**对应主文档**：§15A.6

### 5.2 被 block 的信号后来表现如何？

**检验方法**：从数据库提取所有 `alert_status = "blocked_by_macro"` 的信号，对比后续价格是否触达了 `takeProfit` 或 `stopLoss`。

**预期结果**：被 block 的信号整体胜率应该低于未被 block 的信号。如果被 block 的信号胜率更高，说明 LLM 在误杀好信号——它可能在做超出其认识论能力的判断。

**行动指引**：
- 被 block 信号胜率显著更低 → LLM 外围校验有效，维持现状
- 被 block 信号胜率与正常信号无显著差异 → LLM 贡献不大，可考虑简化
- 被 block 信号胜率更高 → LLM 在误杀，需要审查 Prompt 和判定标准

**对应主文档**：§14.2, §18.4

### 5.3 校准报告建议了什么？系统有没有自动执行？

**检验方法**：检查 `calibration_reports` 表和 `strategyConfig` 的修改记录。

**预期结果**：
1. 当已关闭信号 ≥ `calibrationMinSampleSize` 时，报告给出了具体的阈值调整建议
2. `strategyConfig` 没有在没有人工审核的情况下被修改

**如果不符合**：
- 报告为空或无具体建议 → 校准工作流可能没有真正运行
- 配置被自动修改 → 违反了"不自动改参"约束，系统在做超出授权的事

**对应主文档**：§18.4

### 5.4 回看亏损信号时，因果链上的哪一层判断错了？

**检验方法**：选取所有 `status = "hit_stop"` 的信号，对每一个追溯完整因果链。

**归因检查清单**：

| 因果链层 | 归因问题 | 对应数据 |
|---------|---------|---------|
| 市场状态 | regime 判断是否正确？是否在不可交易状态下放行了？ | `market_contexts.regime`, `regime_confidence` |
| 参与者压力 | 参与者方向判断是否正确？是否在真空期出了信号？ | `market_contexts.participant_bias`, `participant_confidence` |
| 基差背离 | 基差信号是否误导了方向判断？ | `market_contexts.spot_perp_basis`, `basis_divergence` |
| 结构位置 | 入场区域是否合理？止损是否太紧？ | `signals.entry_low`, `entry_high`, `stop_loss` |
| 确认机制 | 确认信号是否真的出现了？还是过早确认？ | `signals.confirmation_status` |
| 语义校验 | LLM 是否应该 block 但没有？ | `macro_assessments.action` |

**预期结果**：大多数亏损信号的根因可以归到因果链上的某一层。

**如果不符合**：如果存在大量"不知道为什么亏了"的信号——因果链上每一层看起来都对，但最终还是亏了——说明系统可能遗漏了某个关键因素，或者某一层的判断标准本身有问题。这种情况需要记录并纳入下一轮校准。

**对应主文档**：§3A, §18.4, §22

### 5.5 按维度分组后，系统是否在某些条件下系统性失效？

**检验方法**：将所有已关闭信号按以下维度分组统计胜率和平均 RR：

| 分组维度 | 目的 |
|---------|------|
| 按 `regime` | 哪种市场状态下系统表现最差？ |
| 按 `liquiditySession` | 哪个时段的信号质量最低？ |
| 按 `structureScore` 分桶 | `minStructureScore` 阈值是否合理？ |
| 按 `confluenceFactors` 数量 | 复合结构是否确实比单一结构表现更好？ |
| 按 `confirmationStatus` | confirmed 信号是否确实优于 pending 信号？ |
| 按 `basisDivergence` | 基差背离是否确实增强了信号质量？ |

**预期结果**：
- 复合结构的胜率 > 单一结构
- confirmed 信号的胜率 > pending 信号
- `london_ramp` 和 `london_ny_overlap` 的信号质量 > `asian_low`
- 有 `basisDivergence` 的信号在对应方向上表现更好

**如果不符合**：
- 复合结构并不优于单一结构 → confluence 加分规则可能需要调整
- confirmed 和 pending 胜率无差别 → 确认机制可能没有实际价值，或确认标准太松
- 亚盘信号质量反而更高 → `sessionDiscountFactor` 可能不合理
- 基差背离没有提升信号质量 → `basisDivergenceThreshold` 可能需要重新校准

**对应主文档**：§18.4（校准维度）

## 6. 评判频率建议

| 阶段 | 评判层 | 频率 |
|------|--------|------|
| 开发阶段 | 第一层（因果链完整性） | 每完成一个 Phase 后执行一次 |
| 测试阶段 | 第二层（行为真实性） | 每次代码变更后通过自动化测试覆盖 |
| 运行阶段（前 3 个月） | 第三层（认知诚实性） | 每周回顾一次 |
| 运行阶段（稳定期） | 第三层（认知诚实性） | 每月回顾一次，每次校准报告生成后必审 |

## 7. 红线规则

以下任一条违反，即判定系统不符合第一性原理：

1. **结构层绕过状态层**：存在任何代码路径让 `detectStructuralSetups()` 在 `regimeConfidence` 低于阈值时仍然运行并产生候选信号
2. **参与者层被跳过**：存在任何代码路径让结构层在没有参与者压力输入的情况下产生候选信号
3. **LLM 进入因果链**：LLM 的输出被用于决定交易方向、入场位、止损位或仓位大小
4. **LLM 升级信号**：语义层把 `watch` 升级为 `standard` 或 `high-conviction`
5. **触碰即触发**：价格进入结构区域后直接输出 `confirmed`，不等待拒绝信号
6. **亏损不可归因**：超过 50% 的 `hit_stop` 信号无法在因果链上定位到具体的判断失误层
7. **自动改参**：系统在没有人工审核的情况下自动修改了 `strategyConfig` 中的任何配置项
8. **研究样本丢失**：被 `block` 的信号没有写入数据库，导致无法事后分析
9. **因果链数据泄露**：`structureScore`、`participantConfidence` 等因果链内部判断数据出现在 LLM Prompt 中
10. **不交易无记录**："跳过信号"的分析周期没有记录跳过原因，无法事后审查

## 8. 与其他文档的关系

- **主架构文档**（`stratum-master-architecture.md`）：定义了系统应该是什么
- **AI 实现规格**（`AI_IMPLEMENTATION_SPEC.md`）：定义了系统应该怎么建
- **阶段文档**（`phases/`）：定义了每一步应该交付什么
- **复审文档**（`reviews/`）：定义了每一步应该检查什么
- **本文档**（`EVALUATION_CRITERIA.md`）：定义了如何判断系统是否真正做到了以上所有文档声称的事情

本文档是最后一道防线。前面的文档可能被正确遵循，但系统仍然可能在运行时表现出非第一性原理的行为——因为规则是死的，市场是活的。只有持续用本文档的标准去检验运行结果，才能确保系统不会在不知不觉中退化。
